# 簡介

容器最迷人的特性，也是最容易讓人一夜白頭的陷阱，是同一件事：容器可以隨時被砍掉、重建、換版、丟棄。前面幾章我們一路把這種「用完即拋」的思維練成肌肉記憶——第 01 章拆解了可寫層的本質，你親眼看過容器停止的那一刻，寫在裡頭的東西是怎麼跟著蒸發的；第 04 章的 `docker diff` 更像一面照妖鏡，把那些在容器裡亂寫檔案、把家當堆在會被拆掉的房子裡的壞習慣，照得無所遁形。

那時埋下的伏筆，這一章要一次收割。

因為現實世界的系統不可能只有「無狀態」這麼乾淨。資料庫裡有訂單、有會員、有一整個事業的命脈；設定檔要能改、日誌要能留、開發時改一行程式碼希望瀏覽器一重整就生效。這些資料不能跟著容器同生共死——它們必須活得比容器更久。這正是本章的核心命題：**如何把資料的去留，徹底管起來。**

答案的主角是 volume。你可以把它想成一個「掛在容器外面的保險箱」：容器是隨時可以拆掉重蓋的房子，重要的家當放進保險箱，房子怎麼拆、怎麼翻新，保險箱始終在原地。這個看似簡單的比喻，背後是一整套設計哲學——資料庫升級變成「砍容器、換映像檔、掛同一個 volume」三步就搞定的例行公事，滾動更新與災難復原的架構也才真正站得住腳。

這一章我們會走過三條掛載的岔路，並且把每一條的脾氣摸透。**volume** 是 Docker 幫你管理、生命週期獨立、可備份可搬家的正規儲存，生產環境的資料一律託付給它；**bind mount** 把主機目錄直接接進容器，主機一改、容器立刻看見，是開發時熱重載與掛設定檔的利器，但也帶著綁死主機路徑、權限打架的原罪；**tmpfs** 則把一塊記憶體當目錄用，容器一停內容就徹底蒸發、永不落磁碟，專門收容那些見不得光的暫時性機密。

光是分清楚三者還不夠。我們會親手做「三代同堂」實驗，讓三個世代的容器、甚至換過映像檔，都證明資料屹立不搖；會拆解「初始填充」這個能救命也能坑人的冷知識，看清 volume 與 bind mount 在此的行為為何恰好相反；會用 PostgreSQL 演一齣完整的「容器升級、資料不動」保衛戰，也順手補上一刀反面教材，讓你看見忘了掛 volume 的那一瞬間，訂單是怎麼隨可寫層一起消失的。接著是 bind mount 開發模式的即改即生效、頭號客訴的 UID 權限難題，多容器共享同一箱資料的正規解法，以及 volume 的備份、還原、跨主機搬家，最後收在清理與治理的斷捨離——畢竟 image 刪了可以重拉，**volume 刪了，資料就是真的沒了。**
