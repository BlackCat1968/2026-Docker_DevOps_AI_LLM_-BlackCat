## 7.1 肥映像檔的三宗罪

先講清楚為什麼值得花力氣瘦身，罪狀條列：

- **拉取慢**：映像檔 1.5GB，一百台主機部署一次就是 150GB 的網路傳輸；擴充副本時新節點光拉映像檔就要等好幾分鐘，尖峰流量早就打完了。
- **攻擊面大**：塞在映像檔裡的編譯器、套件管理器、除錯工具，對服務毫無用處，對入侵者卻是現成軍火庫。掃描器（第 19 章）回報的弱點數量也跟體積成正比。
- **成本高**：Registry 儲存、跨區流量、節點磁碟，全部按位元組計費。瘦身省下的是每個月都在滾的真金白銀。

動刀前先建立量測基線——沒有數字的瘦身都是玄學：

```bash
cd ~/webapp

# 基線三量測:解壓大小、層數、壓縮傳輸量
docker images webapp:1.0 --format '解壓大小: {{.Size}}'
docker history -q webapp:1.0 | wc -l | awk '{print "層數: " $1}'
docker save webapp:1.0 | gzip | wc -c | awk '{printf "壓縮傳輸量: %.1f MB
", $1/1048576}'
```

- 三個數字各對應一種痛：解壓大小吃節點磁碟、層數影響拉取的分段效率、壓縮量決定網路傳輸時間。
- 本章每動一刀就回來量一次，改善幅度用同一把尺說話。

瘦身的總路線圖：

- 核心思想一句話：**FROM 可以寫很多次，每次開一個新階段；最終映像檔只包含最後一個階段的內容**，前面階段用完即棄。
- 過橋的唯一通道是 `COPY --from=階段名`，你決定什麼東西值得裝進箱子。
