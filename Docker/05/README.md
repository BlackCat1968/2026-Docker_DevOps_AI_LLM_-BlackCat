## 動手練習（皆為實務場景）

**練習一：digest 稽核腳本** 場景：資安要求每月盤點生產主機上所有映像檔的 digest 與拉取來源。寫一支 shell 腳本：列出所有映像檔的 `repository:tag`、digest、大小三欄，輸出成 CSV 存檔，檔名帶日期。 提示：`docker images --digests --format` 配 `{{.Repository}},{{.Tag}},{{.Digest}},{{.Size}}`。 驗收標準：CSV 能用試算表開啟，且對照 `docker images` 逐筆一致。

**練習二：層共用量化報告** 場景：主管質疑「容器很吃磁碟」，你要用數字反駁。拉取同一個基底的三個版本標籤（如 python 的三個 3.13 系列 alpine 標籤），記錄：三者邏輯大小總和、`system df` 實際增量、拉取過程 Already exists 的層數。整理成一頁報告說明共用率。 驗收標準：報告包含三個實測數字，並能算出「實際用量占邏輯總和的百分比」。

**練習三：離線交付演練** 場景：下週要去無網路的客戶機房部署。在 A 機（模擬辦公室）save 出服務映像檔、記下 digest；在 B 機（模擬機房，可用 Multipass 另開一台）load 進去，比對兩邊 digest 與層清單完全一致，寫成一頁交付 SOP。 驗收標準：SOP 含指令、驗證方式與「digest 不一致時的處置」，且 B 機的容器能正常啟動。

**練習四：基底遷移影響評估** 場景：資安要求全公司 Python 服務從完整版基底遷到 slim。挑一個帶 C 擴充套件的需求檔（例如含 psycopg2），分別在 python:3.13 與 python:3.13-slim 容器裡 pip install，記錄成功與失敗項目、失敗訊息與需補裝的系統套件清單，寫成遷移風險評估。 驗收標準：評估表列出每個套件在兩種基底的安裝結果，失敗項附上你實測驗證過的解法（如補裝編譯工具或改用二進位輪檔）。

**練習五：偵察機加裝武器** 場景：瘦身偵察機要進排程，主管要它「會自己判斷該不該告警」。擴充 `image_recon.py`：新增 `--threshold` 參數（單位 MB），懸空總量超過門檻時以結束碼 1 退出並印出警告；新增各映像檔「距今幾天沒被任何容器使用」的欄位。 提示：容器使用中的映像檔 ID 用 `client.containers.list(all=True)` 收集比對；建立時間在 `img.attrs["Created"]`。 驗收標準：門檻內結束碼 0、超標結束碼 1，可直接接進 cron 加告警通知。
